# TalentOS テスト実装指示書

## 概要

このファイルはTalentOSのpytestによる自動テストコードを生成するための指示書です。
以下の仕様に従って、テストコードを実装してください。

---

## 前提条件

- FastAPIのTestClientを使用すること（サーバー起動不要）
- テストファイルは機能ごとに分割すること
- 各テストには日本語のdocstringでテスト内容を記載すること
- フィクスチャは `conftest.py` に集約すること
- テスト実行前にDBを初期化し、テスト用データを投入すること

---

## ファイル構成

以下の構成でテストファイルを作成すること：

```
tests/
├── conftest.py           # 共通フィクスチャ・DB初期化
├── test_auth.py          # 認証テスト
├── test_hearing.py       # AIヒアリングテスト
├── test_skillsheet.py    # スキルシートテスト
├── test_search.py        # 人材検索テスト
├── test_users.py         # ユーザー管理テスト
└── test_ui_routing.py    # 画面遷移・ロールアクセステスト
```

### 正常系

```
No.1  engineer01/pass123でログイン → /hearingにリダイレクト
No.2  sales01/pass123でログイン → /searchにリダイレクト
No.3  admin01/admin123でログイン → /usersにリダイレクト

```

### 異常系

```

No.7  正しいIDで誤ったPW → エラーメッセージ・ログインできない
No.8  ID・PW両方空欄で送信 → エラーメッセージ・クラッシュしない
No.9  IDのみ入力・PW空欄で送信 → エラーメッセージ
```

### 境界値

```
No.14 user_idに1000文字の文字列を入力 → クラッシュせずエラー
No.15 PWに1000文字の文字列を入力 → クラッシュせずエラー
No.16 user_idに半角スペースのみ入力 → エラー・ログインできない
No.17 user_idに改行コードを含む文字列 → クラッシュしない・ログインできない
```

### セキュリティ

```

No.23 DevToolsでCookieの値を改ざんしてアクセス → セッション無効・/loginにリダイレクト
No.24 別ブラウザで同じCookieをコピーしてアクセス → セッション無効
No.25 CookieのHttpOnly属性確認 → HttpOnly=Trueになっている
No.26 CookieのSameSite属性確認 → SameSite=Strictになっている
```

## test_hearing.py の実装要件

### 注意事項

Dify APIの呼び出しは必ずモック化すること。
`DIFY_HEARING_API_KEY` を未設定にするか、`unittest.mock.patch` でモック化する。

### 正常系

```
No.1  通常のメッセージを送信 → AIが返答する・theme_completed: false
No.2  複数回やりとりする → 会話履歴がDBに蓄積される
No.3  ページリロード後に会話履歴を取得 → 履歴が復元される
No.4  basic → career に切り替える → 別の会話として扱われる
No.5  career → skills に切り替える → 別の会話として扱われる
No.6  全項目を回答してヒアリングを完了 → theme_completed: true・skill_sheetsに保存される
No.7  ヒアリング途中でページを閉じて再度開く → 途中から続けられる
```

### 異常系

```
No.8  空メッセージで送信 → 送信されない or エラー
No.9  返答が来る前に連続送信 → 二重登録されない
No.10 Dify APIがエラーを返す状態 → エラーメッセージ・クラッシュしない
No.11 Dify APIが完全にダウンしている状態 → エラーメッセージ・クラッシュしない
No.12 ネットワーク切断状態 → エラーメッセージ・クラッシュしない
```

### 境界値

```
No.13 2000文字のメッセージを送信 → クラッシュせず処理される
No.14 絵文字・特殊文字を含むメッセージ → 文字化けしない
No.15 日本語・英語・中国語が混在するメッセージ → 文字化けしない
No.16 タイムアウト（120秒超）のモック → タイムアウトメッセージが返る
No.17 100回以上の会話履歴がある状態 → クラッシュしない
```

### セキュリティ

```
No.18 <script>alert(1)</script> を送信（XSS） → 無害化されて保存される
No.19 ' OR 1=1 -- を送信（SQLi） → DBが破壊されない
No.20 未認証で /api/hearing/chat/stream を叩く → 401
No.21 engineer_idを偽造してAPIを叩く → 403エラー
```

### ロールアクセス制御

```
No.22 salesで /hearing にアクセス → アクセス拒否
No.23 adminで /hearing にアクセス → 正常に表示
No.24 engineerAがengineerBのhearing_logsをAPIで取得 → 403
No.25 engineerAがengineerBのテーマでチャット送信 → 403
```

## テストコード実装時の注意事項

### Dify APIのモック化

```python
# 以下のようにモック化すること
from unittest.mock import patch, AsyncMock

@patch("routers.hearing.DIFY_HEARING_API_KEY", "")
def test_mock_chat():
    # DIFY_HEARING_API_KEYが空の場合、モック動作になる
    ...
```

### 「仕様確認」項目の扱い

以下の項目は仕様が未確定のため、テストコードに `pytest.mark.xfail` を付けて
「仕様確認が必要」というコメントを残すこと：

- 自己PRが79文字の場合の挙動
- 業務内容が79文字の場合の挙動
- 全フィールド空欄で保存した場合の挙動
- PWが1文字の場合の挙動
- 空欄クエリで検索した場合の挙動
- salesのスキルシートアクセス権限

### テスト実行コマンド

```bash
# 全テスト実行
pytest tests/ -v

# 特定ファイルのみ実行
pytest tests/test_auth.py -v

# 仕様確認項目を除いて実行
pytest tests/ -v -m "not xfail"

# 結果をファイルに出力
pytest tests/ -v > test_results.txt
```
