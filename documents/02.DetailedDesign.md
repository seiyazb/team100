# TalentOS 詳細設計書

- **作成日**: 2026年3月11日
- **システム名**: TalentOS（IT人材派遣会社向けスキル管理Webアプリ）
- **技術スタック**: FastAPI + Dify + SQLite + HTML/CSS/JS
- **チーム規模**: 3名

### 1-3. 各画面の概要

#### SCR-01: ログイン画面
- ユーザIDとパスワードを入力してログイン
- 認証成功後、ロールに応じた画面へ自動遷移

#### SCR-02: AIヒアリング・スキルシート作成画面
- 左ペイン：AIとのチャット（ヒアリング）
- 右ペイン：スキルシートのプレビュー（テーマ完了ごとに随時更新）
- ヒアリングテーマ：「基本情報」「経歴」「スキル・資格」の3テーマ
- 全テーマ完了後に「最適化」ボタンを押すとAI最適化が実行される
- **Previewボタン**：押下時にモーダルを表示。スキルシートの整形済みプレビュー（印刷イメージ）を表示し、モーダル内に「PDFダウンロード」ボタンを配置する。PDFはサーバー側で生成（`GET /api/skillsheet/{engineer_id}/pdf`）
- **Proposals for Optimizationパネル**：最適化ワークフロー実行後に表示。AIが「改善できる箇所」を一覧で示す。エンジニアが自分では気づきにくい表現の弱さ・抜け漏れを指摘する役割
- **View in details**：AI最適化提案パネル内のリンク。押下時にモーダルを表示。元の表現・AI推奨表現を比較形式で表示し、「この提案を採用する」ボタンで右ペインのスキルシートに反映する。モーダル内には推奨理由（例：「この表現では採用担当者に伝わりにくいため」）も併せて表示し、エンジニアが内容を理解したうえで採用・却下を判断できるようにする

#### SCR-03: 完成スキルシート画面
- 最適化済みスキルシートの表示
- PDF形式でのエクスポート機能あり
- 管理者は全エンジニアのスキルシートを閲覧・編集可能

#### SCR-04: 人材検索画面
- 自然言語による検索（例：「Python3年以上でAWS経験あり、リモート可能な人」）
- 検索結果をカード形式で一覧表示
- 各カードから対象エンジニアのスキルシートへ遷移可能

#### SCR-05: ユーザ管理画面
- 新規ユーザの登録（氏名・ロール・パスワードの設定）
- DBへのINSERT処理のみ（編集・削除は対象外）

## 2. DB設計（テーブル定義）

### 2-1. ER図（概略）

```
users ──────────── engineers
  │ 1                   │ 1
  │                     │
  │ n                   │ n
hearing_logs      skill_sheets
```

### 2-2. テーブル定義

#### users（ユーザ管理）

| カラム名 | 型 | 制約 | 説明 |
|----------|----|------|------|
| user_id | TEXT | PRIMARY KEY | ユーザID（ログインID） |
| name | TEXT | NOT NULL | 氏名 |
| role | TEXT | NOT NULL | ロール（engineer / sales / admin） |
| password_hash | TEXT | NOT NULL | パスワードのハッシュ値 |

#### engineers（エンジニア基本情報）

| カラム名 | 型 | 制約 | 説明 |
|----------|----|------|------|
| engineer_id | TEXT | PRIMARY KEY, FK(users.user_id) | ユーザIDと一致 |
| full_name | TEXT | | 氏名 |
| birth_year | INTEGER | | 生年 |
| nearest_station | TEXT | | 最寄り駅 |
| remote_available | INTEGER | | リモート可否（0/1） |
| created_at | TEXT | | 登録日時 |

#### skill_sheets（スキルシート）

| カラム名 | 型 | 制約 | 説明 |
|----------|----|------|------|
| sheet_id | TEXT | PRIMARY KEY | スキルシートID（UUID） |
| engineer_id | TEXT | NOT NULL, FK(users.user_id) | エンジニアID |
| theme | TEXT | NOT NULL | テーマ（career / skills / basic） |
| raw_data | TEXT | | ヒアリングで取得した生データ（JSON文字列） |
| optimized_data | TEXT | | AI最適化後のデータ（JSON文字列） |
| created_at | TEXT | | 作成日時 |
| updated_at | TEXT | | 更新日時 |

#### hearing_logs（ヒアリングログ）

| カラム名 | 型 | 制約 | 説明 |
|----------|----|------|------|
| log_id | TEXT | PRIMARY KEY | ログID（UUID） |
| engineer_id | TEXT | NOT NULL, FK(users.user_id) | エンジニアID |
| theme | TEXT | NOT NULL | テーマ（career / skills / basic） |
| messages | TEXT | | 会話履歴（JSON文字列） |
| completed_at | TEXT | | テーマ完了日時（NULL=未完了） |

## 3. API仕様（エンドポイント一覧・入出力）

### 3-1. エンドポイント一覧

| メソッド | エンドポイント | 機能 | 認証 |
|----------|---------------|------|------|
| POST | /api/auth/login | ログイン | 不要 |
| POST | /api/hearing/chat | ヒアリングチャット（1ターン） | 必要 |
| POST | /api/hearing/optimize | スキルシート最適化 | 必要 |
| GET | /api/skillsheet/{engineer_id} | スキルシート取得 | 必要 |
| GET | /api/skillsheet/{engineer_id}/pdf | スキルシートPDFダウンロード | 必要 |
| POST | /api/search | 人材検索 | 必要 |
| POST | /api/admin/users | ユーザ登録 | 必要（adminのみ） |

---

### 3-2. エンドポイント詳細

#### POST /api/auth/login

**リクエスト**
```json
{
  "user_id": "yamada_taro",
  "password": "xxxx"
}
```

**レスポンス（成功）**
```json
{
  "token": "eyJxx...",
  "role": "engineer",
  "name": "山田太郎"
}
```

**レスポンス（失敗）**
```json
{
  "detail": "ユーザIDまたはパスワードが正しくありません"
}
```
- HTTPステータス: 401

---

#### GET /api/skillsheet/{engineer_id}/pdf

**パスパラメータ**
- `engineer_id`: 対象エンジニアのユーザID

**処理概要**
- FastAPI側でSQLiteからスキルシートデータを取得し、サーバー側でPDFを生成して返却する
- PDF生成ライブラリ：`WeasyPrint`または`reportlab`を使用
- 生成したPDFはファイル保存せず、レスポンスとしてそのまま返却する（都度生成）

**レスポンス**
- Content-Type: `application/pdf`
- Content-Disposition: `attachment; filename="skillsheet_{engineer_id}.pdf"`
- ボディ：PDFバイナリ

**エラー時**
```json
{
  "detail": "スキルシートが存在しません"
}
```
- HTTPステータス: 404

## 4. FastAPI ↔ Dify 連携仕様

### 4-1. Difyフロー一覧

| フロー名 | 種類 | 用途 | 優先度 |
|----------|------|------|--------|
| ① ヒアリングチャット | チャットボット | AIによるスキルヒアリング | 最高（3/12〜） |
| ② スキルシート最適化 | ワークフロー | 取得データのAI最適化 | 高（3/13〜） |
| ③ 人材検索クエリ解析 | ワークフロー | 自然言語→検索条件変換 | 中（3/16〜） |

### 4-2. フロー① ヒアリングチャット

**FastAPI → Dify**
```
POST https://dify.example.com/v1/chat-messages
Authorization: Bearer {DIFY_API_KEY_HEARING}

{
  "inputs": { "theme": "career" },
  "query": "テックリードとして8名のチームを...",
  "conversation_id": "abc123",
  "user": "yamada_taro",
  "response_mode": "blocking"
}
```

**Dify → FastAPI（テーマ継続中）**
```json
{
  "answer": "ありがとうございます。期間はどのくらいでしたか？",
  "conversation_id": "abc123",
  "metadata": {
    "theme_completed": false
  }
}
```

**Dify → FastAPI（テーマ完了時）**
```json
{
  "answer": "経歴のヒアリングが完了しました。",
  "conversation_id": "abc123",
  "metadata": {
    "theme_completed": true,
    "theme": "career",
    "extracted_data": {
      "project_name": "金融システムAWS移行",
      "role": "テックリード",
      "team_size": 8,
      "period": "2023/04〜2024/09",
      "tech_stack": ["AWS", "Terraform", "GitHub Actions"],
      "description": "オンプレミスからAWSへの移行をリード"
    }
  }
}
```

**Difyシステムプロンプト（ヒアリング用）**
```
あなたはITエンジニアのスキルシート作成を支援するAIです。
現在のヒアリングテーマ：{theme}

以下の項目を漏れなく聞き取ってください：
{required_fields}

ルール：
- 1度に1〜2つの質問のみ行う
- エンジニアが使う現場固有の用語はそのまま記録する（最適化は後工程で行う）
- 全項目の回答が揃ったら、会話の末尾に以下のJSONブロックを必ず出力する

```json
{
  "theme_completed": true,
  "theme": "{theme}",
  "extracted_data": { ...取得したデータ... }
}
```
```

### 4-4. フロー③ 人材検索クエリ解析

**FastAPI → Dify**
```
POST https://dify.example.com/v1/workflows/run
Authorization: Bearer {DIFY_API_KEY_SEARCH}

{
  "inputs": {
    "search_query": "Python3年以上でAWS経験あり、リモート可能な人"
  },
  "response_mode": "blocking",
  "user": "sales_user"
}
```

**Dify → FastAPI**
```json
{
  "outputs": {
    "conditions": {
      "skills": ["Python", "AWS"],
      "min_experience_years": 3,
      "remote": true
    }
  }
}
```

FastAPIはこの`conditions`をもとにSQLiteを検索し、マッチしたエンジニア一覧を返す。

---

### 4-5. FastAPI vs Dify 責務分担

| 処理 | FastAPI | Dify |
|------|---------|------|
| 認証・ログイン | ✅ DBと照合 | ❌ |
| 会話の転送 | ✅ 橋渡し | ✅ 会話生成 |
| データの保存 | ✅ SQLite | ❌ |
| 項目の抽出 | ❌ | ✅ JSONで返す |
| スキルの最適化 | ❌ | ✅ ワークフロー |
| 検索条件の解析 | ❌ | ✅ ワークフロー |
| DBの検索 | ✅ SQL実行 | ❌ |
